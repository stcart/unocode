# Roadmap разработки сервиса «Практика»

Технологии по ТЗ: **Next.js + TS** (фронт), **Node.js + Express + TS** (бэк), **Prisma + PostgreSQL**, **JWT + bcrypt**, **docxtemplater** для документов, два Docker-контейнера.

Ключевая идея из ТЗ (п.2): разработка **по модулям**, а не «сначала весь бэк, потом весь фронт». То есть для каждого модуля ты делаешь: схема БД → API → интерфейс → проверка руками → следующий модуль. Это и есть твой backlog ниже.

---

## Этап 0. Фундамент (1 неделя)

Прежде чем писать фичи, нужен скелет, на который они лягут.

1. **Инициализация репозитория**: монорепо с двумя папками `frontend/` и `backend/` (можно git submodules не городить, просто одна репа — проще для соло-разработки и для препода).
2. **Backend skeleton**: Express + TypeScript, базовая структура (`src/routes`, `src/controllers`, `src/services`, `src/middleware`), настроить `tsconfig`, `nodemon`/`ts-node-dev` для дев-режима.
3. **Prisma + PostgreSQL**: подними Postgres локально в Docker (`docker-compose.yml` с одним сервисом `postgres` — второй и третий контейнер, для front и back, добавишь позже), настрой `schema.prisma`, прогони первую пустую миграцию, убедись что коннект работает.
4. **Frontend skeleton**: `create-next-app` с TypeScript + `shadcn/ui`, настроить базовый layout, роутинг (App Router), проверить что фронт достучится до бэка (простой `/health` эндпоинт).
5. **Git-flow для себя**: даже соло — заведи ветки по модулям (`feature/cohorts`, `feature/auth` и т.п.) и делай осмысленные коммиты. Это пригодится и для отчёта по практике (карточки задач с гит-ссылками, см. п.10.3 ТЗ), и для защиты — покажешь историю разработки.

**Результат этапа**: пустой, но работающий стек, фронт и бэк общаются, БД подключена.

## Этап 1. Модель данных (по п.11 ТЗ)

Прежде чем писать логику модулей, зафиксируй схему Prisma целиком — она у тебя фактически уже описана аналитиком:

```
User, Cohort, SurveyField, Application, CohortRole,
TestTask, StudentDocumentData, TaskCard
```

Не обязательно сразу продумывать все связи идеально — начнёшь с базовых полей, будешь добавлять по ходу модулей (это нормально при работе по модулям). Но сущности и их ключевые связи (`cohort_id`, `user_id` — везде, где данные должны быть изолированы по когортам, см. п.3 ТЗ) заложи сразу, чтобы не переписывать миграции с нуля.

**Совет**: рисуй ER-диаграмму (хоть на бумаге, хоть в draw.io) до того, как писать `schema.prisma` — сущностей много, связи не тривиальные (например, `StudentDocumentData` — это данные на пересечении `user_id + cohort_id`, не просто на пользователя).

## Этап 2. Модуль «Регистрация и авторизация» (п.5 ТЗ)

Делаешь его вторым, потому что почти все остальные модули требуют `user_id`.

- Backend: `/auth/register`, `/auth/login` — bcrypt для хешей, JWT (access-токен, можно без refresh для MVP — не усложняй), middleware проверки токена.
- Frontend: страницы регистрации/входа, хранение токена (простая cookie или localStorage — для учебного проекта достаточно), защищённые роуты (редирект неавторизованных).

**Результат**: можно зарегистрироваться, войти, получить токен, дёрнуть защищённый эндпоинт.

## Этап 3. Модуль 1 — Когорты, анкета, тестовое (админская часть, п.4 ТЗ)

Это ядро системы — без когорты не работает вообще ничего (заявки, документы, задачи — всё привязано к когорте).

Разбей на подзадачи:
1. **CRUD когорт** (создание, список, редактирование) — только для админа. Поля: название/год, даты приёма заявок, даты практики.
2. **Конструктор анкеты**: список полей (`text` / `select`), привязанных к когорте, с возможностью добавить/убрать поле при создании новой когорты (структура переиспользуется год от года — не надо делать сложное версионирование).
3. **Роли/треки когорты** (`CohortRole`) — простой список строк на когорту.
4. **Тестовое задание**: текстовое поле-контент + `published_at`. Логика «кандидату видно только после отправки анкеты» реализуется на уровне фронта/API — не отдавай контент тестового, пока по этому `user_id` нет заявки.

**Проверка этапа**: админ может создать когорту «2026», настроить под неё анкету и роли.

## Этап 4. Модуль 3 — Заявка на практику (п.6 ТЗ)

- Публичная страница анкеты по ссылке когорты (доступна неавторизованным — п.3 ТЗ), но отправка заявки требует регистрации/входа (сценарий из п.5: сначала регистрация, потом заявка привязывается к `user_id`).
- Логика автоподстановки: если у пользователя уже есть данные из прошлой заявки (другая когорта) — подтягивай их в новую форму как значения по умолчанию, разрешай редактировать.
- Статусы заявки: `pending / approved / rejected` + комментарий при отклонении.
- После отправки анкеты — открой пользователю доступ к тестовому заданию (если оно опубликовано), иначе — заглушка «появится позже» + заготовка под email-уведомление (само письмо можно сделать заглушкой/логом в MVP, п.13 ТЗ прямо говорит, что e-mail-сценарии кроме публикации теста — под вопросом).

## Этап 5. Личный кабинет практиканта — каркас (п.7 ТЗ)

Здесь просто собираешь скелет из трёх вкладок:
- **Заявки** — список своих заявок по всем когортам + статусы (данные уже есть с этапа 4).
- **Документы** — заглушка, наполнишь в этапе 6.
- **Задачи** — заглушка, наполнишь в этапе 7, появляется только после одобрения заявки в текущей когорте.

Смысл сделать каркас пораньше — сразу видно, куда «вешать» следующие модули, и есть на чём показывать прогресс на созвонах с руководителем.

## Этап 6. Модуль 5 — Документы (п.8 ТЗ) — самый трудоёмкий

Разбей на шаги:

1. **Сущность `StudentDocumentData`**: одна запись на `user_id + cohort_id`, поля переиспользуются между документами (не дублируй `student_fio` в трёх местах).
2. **Форма заполнения полей** во вкладке «Документы»: автосохранение по потере фокуса (auto-binding) + кнопка «Сохранить» — сохраняй в БД сразу, не только на фронте.
3. **Условия доступности** по таблице из п.8.3 — реализуй как чистую функцию `canGenerate(document, data, application, cohort)`, которая проверяет обязательные поля + доп.условие (заявка одобрена / отзыв написан админом / отчёт загружен и одобрен). Она пригодится и на фронте (активность кнопки), и на бэке (защита эндпоинта генерации).
4. **Шаблоны .docx с плейсхолдерами `{{field}}`**: попроси у заказчика/аналитика реальные .docx-шаблоны (ИЗ, отзыв, титульный лист) — без них не протестируешь docxtemplater. Подставляй туда placeholder'ы из таблицы п.8.1.
5. **Генерация через docxtemplater**: эндпоинт `/documents/:type/generate` — берёт данные из `StudentDocumentData` + статичные поля когорты (`practice_start/end`, город/год) + захардкоженные статичные поля (место практики, руководители — см. таблицу констант в ТЗ) → рендерит .docx → отдаёт файл на скачивание.
6. **Загрузка отчёта** (п.8.4): отдельный аплоад-эндпоинт (файл на диск или в БД как blob — для учебного проекта локальное хранилище на диске backend-контейнера вполне ок), поле `report_file_url` + `report_admin_approved`.

**Совет по порядку**: делай документы в порядке зависимости — сначала ИЗ (не требует админа, только одобренная заявка), потом Отзыв (требует, чтобы админ сначала заполнил `review_*` — значит, тебе всё равно понадобится кусок админ-панели из этапа 7), потом Титульный лист (требует загрузки отчёта + одобрения админом).

## Этап 7. Модуль 6 — Админ-панель (п.9 ТЗ)

Собирается параллельно с этапом 6, так как «Отзыв» и «Титульный лист» зависят от действий админа.

- **Вкладка «Заявки»**: список заявок когорты, просмотр анкеты, одобрить/отклонить + комментарий, назначение роли при одобрении.
- **Вкладка «Документы»**: список практикантов + 3 колонки-индикатора (готовы ли поля со стороны студента), форма написания отзыва (`review_*` поля), открытие отчёта + отметка допуска к скачиванию.
- **Вкладка «Задачи»**: переиспользуешь компонент сетки задач из этапа 8, просто без переключателя «свои/все» — админ сразу видит всех.
- **Управление когортой**: сюда просто выносишь UI из этапа 3 (создание когорты, анкеты, ролей, тестового) в единый раздел админки.

## Этап 8. Модуль 7 — Задачи и прогресс (п.10 ТЗ)

Самый «UI-интенсивный» модуль:

1. Сетка по неделям, построенная от `practice_start` до `practice_end` когорты, без сб/вс — напиши утилитку генерации рабочих дней между датами один раз, переиспользуй и на фронте (рендер сетки), и при желании на бэке (валидация дат карточки).
2. Пролистывание недель (не весь срок одним списком) — обычная пагинация по неделям с текущей неделей по умолчанию.
3. Чекбокс «Показать всех» — переключение между «мои карточки» и «карточки всех участников когорты» (рядом с ФИО — роль из `CohortRole`).
4. Карточка задачи (клик по ячейке дня): модалка с полями `title/description`, `artifact_link`, отображением `updated_at`. Практикант создаёт карточки сам.

Этот модуль неплохо ложится на **Figma-скетч перед кодом** — раз ты уже работал со Scripter/UI-генерацией, набросай сетку недель и модалку в Figma до вёрстки, чтобы не переделывать сетку в React несколько раз.

## Этап 9. Нефункциональные требования и докеризация (п.12 ТЗ)

Оставляй на конец, но не забывай:
- `Dockerfile` для frontend и backend отдельно + `docker-compose.yml`, объединяющий их с Postgres.
- Проверь на пустой машине/чистом окружении, что `docker-compose up` поднимает всё с нуля (миграции Prisma должны прогоняться автоматически при старте бэкенда).
- Убедись, что нет сценариев потери данных при обновлении страницы (автосохранение из этапа 6 — самое уязвимое место).

## Этап 10. Полировка и защита

- Пройдись по открытым вопросам из п.13 ТЗ — по каждому либо прими решение сам и зафиксируй его (для защиты полезно явно проговорить: «я решил X, потому что Y»), либо вынеси как заведомое ограничение MVP.
