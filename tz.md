# Техническое задание
## Сервис для организации и сопровождения практики

## 1. Общие сведения

### 1.1. Название системы
«Практика» (рабочее название, подлежит уточнению) — веб-сервис для организации приёма студентов на практику, ведения их заявок, документооборота и контроля выполнения задач в рамках практики.

### 1.2. Цель системы
Автоматизировать и перевести в единую систему процесс, который сейчас ведётся вручную (заявки через почту/личные сообщения, документы — отдельными файлами, контроль задач — на созвонах и в чатах):

- приём заявок на практику через публичную форму;
- отбор кандидатов (анкета + тестовое задание);
- автоматическое формирование документов практики (индивидуальное задание, отзыв, титульный лист отчёта);
- отслеживание выполнения задач/прогресса практикантов по неделям;
- администрирование всего процесса по годовым потокам (когортам).

### 1.3. Пользователи системы

Роли:

- **Практикант** — подаёт заявку, проходит практику, формирует документы, отчитывается по задачам.
- **Админ** — единая роль с полным доступом: создаёт когорты, анкеты, тестовые задания, рассматривает заявки, пишет отзывы, работает с документами и задачами всех практикантов. Отдельной урезанной роли «руководитель практики» в MVP нет — эти функции входят в роль админа.

## 2. Архитектура и технологический стек

Система реализуется как монолит из двух контейнеров. Frontend и backend — отдельные контейнеры, общаются через REST API. Каждый модуль системы реализуется совместно на фронте и бэке (сквозная разработка «по модулям», а не «сначала весь бэк, потом весь фронт»).

| Компонент | Технология |
|---|---|
| Frontend | Next.js, TypeScript, UI-компоненты — shadcn/ui |
| Backend | Node.js, TypeScript, Express |
| ORM | Prisma |
| База данных | PostgreSQL |
| Формирование документов | docxtemplater (шаблоны .docx с плейсхолдерами вида `{{поле}}`) |
| Кэширование | не требуется |

## 3. Ролевая модель и разграничение доступа

- **Неавторизованный пользователь**: видит только публичную анкету по ссылке текущей активной когорты.
- **Практикант (авторизованный)**: видит и редактирует только свои данные — свои заявки, свои документы, свои задачи.
- **Админ**: после входа выбирает когорту («проваливается» в неё) и работает в контексте этой одной когорты — видит все заявки, все документы, все задачи участников именно этой когорты. Данные разных когорт между собой не пересекаются.

## 4. Модуль 1. Когорты, анкета и тестовое задание (админская часть)

### 4.1. Когорта

Когорта — это один поток практики (например, «2026», «2027»). У каждой когорты:

- уникальное название/год;
- даты начала и окончания практики (используются далее в модуле «Задачи» для построения сетки по неделям, а также автоматически подставляются в документы — см. п. 8);
- даты начала и окончания приёма заявок;
- своя анкета, своё тестовое задание;
- свой набор ролей/треков (см. п. 4.3);
- свой список заявок, документов, задач — изолированно от других когорт.

Админ может создать новую когорту в любой момент (практика проводится примерно 2 раза в год).

### 4.2. Анкета

Анкета привязана к когорте и доступна кандидату по публичной ссылке (без авторизации).

Анкета состоит из набора полей. Тип поля — один из двух:

- текстовое поле (короткий/длинный текст);
- выбор из списка (select/радиокнопки) — например, для вопросов про роль, стек технологий и т.п.

Решение по итогам уточнения: анкета не версионируется по содержанию год от года — структура вопросов переиспользуется одна и та же (в 2026, 2027 и далее), но админ должен иметь возможность отредактировать список вопросов при создании новой когорты (добавить/убрать поле), если потребуется.

Поле «дата рождения» и подобные «профильные» поля не включаются — анкета является только начальным этапом согласования заявки на практику, интересуют базовые вопросы: ФИО, группа/курс, желаемая роль, стек и т.п. (финальный перечень вопросов — см. открытые вопросы, п. 13).

### 4.3. Роли/треки в когорте

Отдельная сущность в БД: список ролей, актуальных для конкретной когорты (например, «Frontend», «Backend», «Аналитик» — произвольный текст, задаётся админом заново под каждую когорту).

Роль назначается практиканту (админом) после одобрения заявки и в дальнейшем отображается рядом с его ФИО в модуле задач.

### 4.4. Тестовое задание

Тестовое задание также привязано к когорте.

Кандидату тестовое задание становится видно только после отправки анкеты (не раньше).

Пока тестовое задание не опубликовано админом — кандидат видит уведомление, что задание появится позже, и получает уведомление на e-mail в момент публикации.

## 5. Модуль 2. Регистрация и авторизация

- Классическая схема: email + пароль.
- Пароли хранятся в виде bcrypt-хешей.
- Авторизация — через JWT.
- Сценарий: пользователь сначала регистрируется в системе (email/пароль), затем подаёт заявку — заявка сразу привязывается к его учётной записи (`user_id`). Это позволяет заходить с разных устройств и видеть свои заявки в личном кабинете.

## 6. Модуль 3. Заявка на практику

Заявка = заполненная анкета конкретной когорты, привязанная к пользователю.

Если пользователь подаёт заявку повторно (например, был на практике на 2 курсе, подаёт заявку на 3 курсе) — уже известные системе данные (из прошлой заявки/профиля) должны автоматически подставляться в новую анкету, чтобы не заполнять их заново. Пользователь может их скорректировать при необходимости.

У заявки есть статус: на рассмотрении / одобрена / отклонена.

При отклонении админ может оставить комментарий (см. п. 9.1).

## 7. Модуль 4. Личный кабинет практиканта

Практикант в личном кабинете видит:

- Вкладка **«Заявки»** — список своих заявок по всем когортам, в которых он участвовал/участвует, и их статусы.
- Вкладка **«Документы»** — см. модуль 5.
- Вкладка **«Задачи»** — см. модуль 7 (только после одобрения заявки в текущей когорте).

## 8. Модуль 5. Документы

Три типа документов практики, каждый формируется автоматически из шаблона .docx (библиотека docxtemplater, подстановка значений в плейсхолдеры `{{...}}`). Подписи в документах не проставляются автоматически (руками, после выгрузки).

### 8.1. Состав полей документов

**Статичные поля** (одинаковые для всех, «зашиты» в шаблон .docx без возможности редактирования в MVP):

| Поле | Значение |
|---|---|
| Институт | ИРИТ-РТФ |
| Кафедра | школа бакалавриата (школа) |
| Место прохождения практики | ИП Езуб Антон Сергеевич, удалённое прохождение |
| Вид практики | Производственная практика |
| Тип практики | Производственная практика, технологическая |
| Содержание отчёта (в ИЗ) | введение, описание работы, результат практики, заключение, список использованных источников |
| Организационный / заключительный этапы графика (в ИЗ) | фиксированный текст |
| Руководитель практики от предприятия | Езуб Антон Сергеевич |
| Руководитель практики от УрФУ | Корнякова Елена Михайловна |
| Город, год | Екатеринбург, [год когорты] |

**Переменные поля, заполняемые студентом** (хранятся в БД в единой сущности «данные документов студента» на когорту — переиспользуются между документами, где совпадают):

| Плейсхолдер | Описание | ИЗ | Титул | Отзыв |
|---|---|---|---|---|
| student_fio | ФИО студента | ✓ | ✓ | ✓ |
| group | Группа (напр. РИ-330930) | ✓ | ✓ | ✓ |
| direction_code | Код направления (напр. 09.03.03) | ✓ | — | — |
| direction_name | Наименование направления (напр. Прикладная информатика) | ✓ | — | — |
| program_name | Наименование образовательной программы | ✓ | — | — |
| specialty | Специальность (направление подготовки) для титула | — | ✓ | — |
| practice_topic | Тема задания на практику | ✓ | ✓ | — |
| main_stage_tasks | Перечень работ основного этапа практики (список задач) | ✓ | — | — |

**Поля, подставляемые автоматически из настроек когорты** (без ввода пользователем):

| Плейсхолдер | Описание | ИЗ | Отзыв |
|---|---|---|---|
| practice_start | Дата начала практики | ✓ | ✓ |
| practice_end | Дата окончания практики | ✓ | ✓ |

**Переменные поля, заполняемые админом** (только для документа «Отзыв»):

| Плейсхолдер | Описание |
|---|---|
| review_activities | Мероприятия, осуществлённые студентом за время практики (текст) |
| review_characteristic | Краткая характеристика уровня подготовки и отношения к работе (текст) |
| review_employed | Был ли студент трудоустроен на время практики (да/нет + должность) |
| review_next_practice | Предложено ли пройти следующую практику в организации (да/нет) |
| review_employment_offer | Предложено ли трудоустройство после завершения обучения (да/нет) |
| review_suggestions | Предложения и замечания от организации (свободная форма) |
| review_grade | Оценка за практику |

**Примечание по архитектуре данных**: поля `student_fio`, `group`, `practice_topic` и т.п. переиспользуются в нескольких документах — хранить их нужно один раз на уровне «данные документов студента в рамках когорты», а не дублировать отдельно для каждого документа. Поля `practice_start`/`practice_end` не хранятся у студента — берутся напрямую из модели когорты.

### 8.2. Логика заполнения полей

Все заполненные студентом и админом поля сохраняются в БД (отдельная сущность документов пользователя в рамках когорты), а не только на фронте — обновление страницы не должно приводить к потере данных. Сохранение — либо по кнопке «Сохранить», либо автосохранением при потере фокуса поля (auto-binding).

Как только все обязательные поля конкретного документа заполнены — кнопка «Сформировать документ» становится активной. До этого момента кнопка неактивна/скрыта.

### 8.3. Условия доступности каждого документа

| Документ | Обязательные поля студента | Дополнительное условие активации кнопки «Сформировать» |
|---|---|---|
| Индивидуальное задание (ИЗ) | student_fio, group, direction_code, direction_name, program_name, practice_topic, main_stage_tasks | Заявка практиканта одобрена. Формируется самостоятельно, без участия админа. |
| Отзыв о практике | student_fio, group (для подстановки данных студента) | Все поля review_* заполнены админом. Практикант не может сформировать документ, пока отзыв не написан админом. |
| Титульный лист отчёта | student_fio, group, specialty, practice_topic (уже будут заполнены при формировании ИЗ) | Практикант должен сначала загрузить сам отчёт о практике (файл, см. п. 8.4), и админ должен ознакомиться с отчётом и поставить отметку «можно скачивать» — только после этого кнопка становится активна у практиканта. |

### 8.4. Загрузка отчёта о практике

Отдельная функция в личном кабинете практиканта: загрузка файла отчёта (docx/pdf) — прикреплённый файл, не ссылка.

Загруженный отчёт становится виден админу во вкладке «Документы» админ-панели (см. п. 9.2) — с возможностью открыть/ознакомиться и поставить отметку о допуске к скачиванию титульного листа.

## 9. Модуль 6. Админ-панель

Работает в контексте одной выбранной когорты.

### 9.1. Вкладка «Заявки»

- Список всех заявок в рамках когорты (в отличие от практиканта, который видит только свою).
- Просмотр анкеты кандидата.
- Действия: одобрить / отклонить (с возможностью оставить комментарий при отклонении).
- После одобрения — назначение роли/трека практиканту из списка ролей когорты (п. 4.3).

### 9.2. Вкладка «Документы»

- Список ФИО всех практикантов когорты с одобренной заявкой.
- Три столбца-индикатора (галочка/крестик) по каждому документу: ИЗ, Отзыв, Титульный лист — отражают, заполнил ли практикант все нужные поля со своей стороны.
- Действия админа:
  - написать текст отзыва, заполнив поля review_activities, review_characteristic, review_employed, review_next_practice, review_employment_offer, review_suggestions, review_grade (открывает возможность практиканту сформировать документ «Отзыв»);
  - открыть загруженный отчёт практиканта и поставить отметку «допущен к скачиванию титульного листа».

### 9.3. Вкладка «Задачи»

Список всех практикантов когорты и их карточек задач по неделям (аналог того же интерфейса, что у практиканта, но сразу для всех, без переключения галочкой).

### 9.4. Управление когортой

Создание новой когорты, настройка анкеты, тестового задания, ролей/треков, сроков подачи заявок и сроков практики (см. п. 4).

## 10. Модуль 7. Задачи и прогресс

### 10.1. Общая логика

Это не канбан-доска — карточки не перетаскиваются, это фиксированная сетка «ячеек» для заполнения информации.

Сетка строится по датам в рамках срока практики текущей когорты (даты старта/окончания заданы в настройках когорты), без учёта суббот и воскресений.

Отображение — по неделям, с возможностью пролистывать текущую/соседние недели (не показывать весь срок практики одним длинным списком).

### 10.2. Видимость

По умолчанию практикант видит только свои задачи.

Чекбокс «Показать всех» — переключает отображение на список всех участников когорты (аналогично тому, что админ видит по умолчанию).

Разделения по трекам нет: рядом с ФИО каждого участника просто подписывается его роль (из п. 4.3).

### 10.3. Карточка задачи

Клик/плюсик по ячейке дня открывает всплывающую карточку со следующими полями:

- название/описание задачи;
- что было сделано (текст);
- ссылка на артефакт (единое поле-ссылка: GitHub, Figma, скриншоты и т.п. — без файлового хранилища, только внешняя ссылка).

Практикант может самостоятельно создавать и заполнять карточки — не обязан ждать, пока задачу поставит «проджект»/куратор.

При сохранении/изменении карточки фиксируется время последнего обновления (`updated_at`) — оно должно быть видно в интерфейсе, чтобы можно было отследить, кто и когда отчитался.

## 12. Нефункциональные требования

- Frontend и backend — в отдельных Docker-контейнерах.
- Хранение данных — PostgreSQL, доступ через Prisma ORM.
- Высокая нагрузка не ожидается, кэширование не требуется.
- Все пользовательские данные (поля анкеты, поля документов, карточки задач) должны сохраняться в БД сразу/по явному сохранению — потеря данных при обновлении страницы недопустима.

## 13. Вне рамок MVP / уточнить отдельно

- Точный перечень вопросов анкеты — структура определена (текстовые поля и select, единая анкета год от года с возможностью правки при создании когорты), финальный список конкретных вопросов (ФИО, группа/курс, желаемая роль, стек и т.п.) нужно согласовать отдельным списком.
- Формат e-mail-уведомлений — подтверждён только сценарий публикации тестового задания; нужно решить, требуются ли уведомления на остальных шагах (одобрение/отклонение заявки, готовность отзыва, допуск к скачиванию титульного листа).
- Место прохождения практики в документах — на MVP статично («ИП Езуб Антон Сергеевич, удалённое прохождение», «зашито» одинаково для всех); нужно решить, задаётся оно глобальной настройкой или настройкой на уровне когорты (на случай, если места будут отличаться между потоками).
- Поле specialty на титульном листе — по факту дублирует связку direction_code + direction_name из ИЗ; нужно решить, автоподставлять его из этих полей или оставить отдельным полем ввода (на случай расхождений в формулировках между документами).
