# Проект «Практика»

Веб-сервис для организации и сопровождения практики студентов. Полное ТЗ — в `tz.md`, план разработки по этапам — в `roadmap-praktika.md`. Перед началом работы над любой задачей сверяйся с обоими файлами.

## Как мы работаем

- Разработка ведётся **строго по этапам roadmap-praktika.md**, один этап за раз. Не забегай вперёд и не реализуй функциональность из будущих этапов, даже если она кажется логичным продолжением.
- Внутри этапа — **сквозная разработка по модулям**: для одной фичи сразу делаем схему БД (если нужно) → API → интерфейс, не растягиваем на "сначала весь бэк".
- Я — единственный разработчик, работаю один. Не предлагай разделение работы "на бэкендера/фронтендера/девопса" — весь код пишу и ревьюю я сам через тебя.
- Перед крупными изменениями — кратко опиши план (какие файлы создашь/изменишь), не пиши сразу тысячи строк без объяснения.

## Стек (менять без обсуждения нельзя)

- **Frontend**: Next.js (App Router) + TypeScript + shadcn/ui
- **Backend**: Node.js + Express + TypeScript
- **ORM / БД**: Prisma + PostgreSQL
- **Auth**: email/password, bcrypt для хешей, JWT для авторизации
- **Документы**: docxtemplater (шаблоны .docx с плейсхолдерами `{{field}}`)
- **Инфраструктура**: frontend и backend — отдельные Docker-контейнеры, общаются через REST API

## Правила по коду

- Весь новый код — TypeScript, без `any` без крайней необходимости.
- Изоляция данных по когортам — критичное правило: почти каждый запрос к БД, связанный с заявками/документами/задачами, должен фильтроваться по `cohort_id`. Не забывай про это при написании контроллеров.
- Роли в системе — `Практикант` и `Админ` (единая роль с полным доступом, без отдельной "руководитель практики" в MVP). Проверка роли — через middleware, не разбрасывай проверки `if (user.role === 'admin')` по контроллерам без необходимости.
- Данные форм (анкета, документы, карточки задач) должны сохраняться в БД сразу или по явному сохранению — потеря данных при обновлении страницы недопустима (см. п.12 tz.md).
- Не выдумывай функциональность, которой нет в tz.md. Если нужно решение по открытому вопросу (см. п.13 tz.md) — сначала спроси меня, не выбирай самостоятельно молча.

## Чего не делать

- Не вводи Redis/кэширование — явно не требуется (п.12 tz.md).
- Не делай отдельную роль "руководитель практики" — она входит в роль админа.
- Не версионируй анкету по годам как отдельную сущность-историю — структура анкеты переиспользуется, просто редактируется при создании когорты.
- Не храни файлы отчётов как ссылки — только реальная загрузка файла (п.8.4 tz.md).
